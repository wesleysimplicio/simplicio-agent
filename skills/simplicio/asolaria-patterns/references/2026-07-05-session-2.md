## Cosign Chain — SQLite Persistence

The cosign chain (`src/asolaria/cosign_chain.rs`) was originally a scaffold (v0.1) with in-memory storage. This session added:

**SQLite persistence:**
- `load_or_create()` — reads last row from SQLite, computes its sha16 as head
- `append(&row)` — validates monotonic row, chain link, kind, then INSERTs into SQLite
- `verify()` — SELECTs all rows ordered by row ASC, re-derives sha16 chain, returns count

**Critical bug fixed:** `load_or_create()` was using `prev_sha16` of the last row as head, but head should be the sha16 OF the last row. Fixed by computing `sha16_of("row={rnum}|ts={ts}|prev={prev}|kind={kind}|payload={payload}")` from the last row's fields.

**SQLite schema:**
```sql
CREATE TABLE IF NOT EXISTS cosign_chain (
    row INTEGER PRIMARY KEY,
    ts_ns INTEGER NOT NULL,
    prev_sha16 TEXT NOT NULL,
    kind TEXT NOT NULL,
    payload_sha16 TEXT NOT NULL,
    sig BLOB NOT NULL
);
```

**Integration with hookwall:** Each verdict now creates a `CosignRow` and appends to the chain.

## simplicio edit — JSON Plan Format (learned via trial and error)

The correct format for `simplicio edit --plan <file>`:

```json
{
  "file": "src/agent_state_command.rs",
  "operations": [
    {
      "op": "replace",
      "find": "exact text to find (unique in file)",
      "with": "replacement text"
    }
  ]
}
```

Keys discovered:
- `op` field is REQUIRED and must be `"replace"`
- `find` is the text to find (NOT `old_string`)
- `with` is the replacement (NOT `new_string` or `replace`)
- The file path is under `"file"` at the top level, NOT inside operations
- Each operation is a JSON object in the `"operations"` array
- SHA tracking: if a file's SHA changes between edits, subsequent `find` patterns fail. Serialize dependent edits.

## Build Release Notes

- `cargo build` (debug) takes 1-3 minutes
- `cargo check` takes 20-60 seconds
- `cargo clean` freed 20.3GB

## Record-Task Persistence

`cmd_record_task` in `agent_state_command.rs`:
1. Loads worker from SQLite
2. Calls `update_worker_task(pid, success, duration_ms)` which runs:
   ```sql
   UPDATE agent_workers SET
       task_count = task_count + 1,
       success_count = CASE WHEN ?1 THEN success_count + 1 ELSE success_count END,
       total_duration_ms = total_duration_ms + ?2,
       state = 'Heartbeating',
       last_heartbeat = ?3
   WHERE pid = ?4
   ```
3. Executes watcher verification
4. Reports result

## Pitfalls

- **Don't use `args.get(N)` for subcommand dispatch** when the command name is consumed by the outer dispatch. Use `args.first()` instead.
- **Each simplicio CLI invocation is a new process** — state doesn't persist between calls except through SQLite. Each call creates a fresh `load_or_create()`.
- **Cosign chain verify fails** if `load_or_create` returns wrong head. Always compute head as SHA of the LAST row, not its prev_sha16.
- **Killed: 9** usually means OOM on macOS after long builds. Wait for memory to recover before running the binary.
- **`<` in Rust expressions** can be parsed as generic parameter start. Wrap comparisons in parens: `(a as f64 / b as f64) < 0.3`.
- **`unwrap_or(&"".to_string())`** creates a temporary that's dropped. Use a `let empty = String::new();` variable first.
- **`query_row` returns `Result<T, Error>`, not `Result<Option<T>, Error>`**. Don't wrap in `Ok(Some(...))`.
