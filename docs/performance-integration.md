# Performance integration manifest

`tools/perf_integration_manifest.py` produces the deterministic
`simplicio.perf-integration-manifest/v1` document used to audit the shipped
performance paths. It is offline: no provider calls, credentials, timestamps,
host paths, or dependency versions are included in the output.

Generate a report for the current checkout:

```bash
python tools/perf_integration_manifest.py --json
```

Generate the committed fixture (or another machine-readable artifact):

```bash
python tools/perf_integration_manifest.py \
  --generate fixtures/bench/perf/perf-integration-manifest.v1.json
```

Validate an artifact and its source hashes against a checkout:

```bash
python tools/perf_integration_manifest.py \
  --validate fixtures/bench/perf/perf-integration-manifest.v1.json
```

## v1 shape

The top-level object has `schema`, `version`, `generator`, `axes`, and a
deterministic `summary`. Each axis records `source` and `source_sha256`,
`call_sites`, `config`, and `fallback` metadata. The source paths are
repository-relative POSIX paths, so the same checkout produces byte-stable
JSON on different machines.

Every axis classifies all nine stages independently:

`PRESENT`, `SAME_SOURCE`, `BUILT`, `PACKAGED`, `INSTALLED`, `INVOKED`, `E2E`,
`DEFAULT`, and `GATED`.

The canonical v1 result is `stage_results[STAGE]`, with a `status` of `pass`,
`fail`, `unknown`, or `not_applicable`, plus optional reason/evidence. A
`not_applicable` stage is successful for the axis (for example uvloop on
Windows, where the declared asyncio fallback is the supported path). The
`stage_status` map is a compact classification view; `stages` is a legacy
boolean compatibility view.

The fixture is intentionally generated by the same command used in CI. If a
source file, call site, package extra, fallback, or doctor gate changes, the
validator reports the stale SHA or stage instead of silently accepting a
regression.
