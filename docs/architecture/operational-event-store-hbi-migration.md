# Operational event-store migration boundary

This document describes the bounded issue #517 slice for the Agent's
`AwarenessReceipt` event store. It is an opt-in migration path; the existing
`OperationalEventStore` JSONL writer and reader remain unchanged until a
separately reviewed cutover.

## Boundary

`migrate_legacy_event_store()` reads the complete legacy JSONL file, validates
each line with the existing `AwarenessReceipt` and `OperationalScope` types,
and publishes a local HBP chain plus its HBI offset sidecar. The structured
receipt is carried in the existing `tools.hbi.pack_binary()` representation and
base64-encoded inside an HBP field so pipe, newline, and arbitrary value data
cannot change row framing.

The target consists of:

- `<target>.hbp`: hash-chained `AWARENESS` rows;
- `<target>.hbi`: byte offsets generated by `HbpLedger`;
- `<target>.migration`: non-JSON source/target digest metadata;
- `<target>.migration.pending`: a short-lived transaction marker used to
  recover an interrupted publication.

## Guarantees in this slice

- The legacy input is bounded and fully parsed before target publication.
- Corrupt, truncated, invalid-UTF-8, or scope-invalid input leaves no target.
- Temporary HBP/HBI files are verified for chain integrity, offsets, and
  `AwarenessReceipt` round-trip equivalence before replacement.
- A failure between the HBP and HBI replacements removes the partial target;
  a retry also cleans a pending publication left by process interruption.
- A matching source digest and verified target make a second invocation a
  no-op. An unmarked existing target is never overwritten.

## Explicit non-claim

The HBP/HBI files and marker are a local Agent compatibility boundary. This
slice does **not** claim conformance with Runtime HBI v1, cross-repository
cutover, removal of the legacy JSONL writer, or issue-wide JSON elimination.
Those remain separately owned and `UNVERIFIED` until Runtime contracts,
reader/writer cutover, package scans, and end-to-end evidence are available.
